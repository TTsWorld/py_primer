# MVP 定义：AI 辅助历史研究时间线工作台

**日期**：2026-03-18  
**状态**：Draft v0.1  
**相关材料**：

- [现有技术调研](./01-investigate.md)

## 1. 一句话定义

第一版产品不是“知识管理平台”，而是一个 `单用户、时间线优先、可交互` 的研究工作台。

它要解决的问题是：

> 把一篇或多篇线性文本资料，半自动转换成一条 `可编辑、可折叠、可追问、可回溯证据` 的时间线，帮助用户更快看清历史发展脉络。

---

## 2. 当前默认假设

这份 MVP 文档基于以下假设起草：

1. 第一版优先服务 `你自己`，不是多人协作产品。
2. 第一版优先支持 `历史类研究`，尤其是“梳理发展脉络”场景。
3. 第一版的主视图只有一个：`时间线`。
4. 第一版允许有辅助信息面板，但不要求完整关系图、导图、地图等多视图。
5. 第一版先验证“研究效率是否显著提升”，而不是“自动化程度是否最高”。

如果后续目标从“个人研究工作台”切换成“面向他人的产品”，这份文档需要再做一轮收敛。

---

## 3. 产品目标

### 3.1 要达成的事

MVP 要让用户完成下面这个闭环：

1. 导入一篇资料。
2. 自动抽取其中的时间、事件、人物、概念。
3. 生成一条主时间线。
4. 把次要背景、补充说明、争议点挂载在主线上，并默认折叠。
5. 点击任一节点，可以看到证据原文、出处和 AI 解释。
6. 用户可以修正时间、标题、重要度和挂载关系。
7. 用户可以围绕当前节点继续追问，补充更多上下文。

### 3.2 不追求的事

第一版不追求：

- 自动生成所有图形视图
- 全自动、零校正的高精度历史建模
- 多用户协作
- 复杂权限系统
- 公开发布与分享能力
- 大规模资料库管理
- 完整的通用知识图谱平台

---

## 4. 目标用户与核心场景

### 4.1 目标用户

第一阶段的目标用户就是你这种研究型学习者：

- 会阅读书籍、博客、文章、对话记录
- 不满足于线性阅读
- 需要围绕一个主题不断补线索
- 愿意做人工校正，但不想重复做机械整理

### 4.2 核心场景

以当前 [article.md](../../article.md) 为例，用户希望：

- 从一篇“AI 发展脉络”草稿里，快速抽出关键历史节点
- 按时间排序形成主线
- 把背景解释、争论、补充人物挂到主线下面
- 默认只看核心事件，需要时再展开细节
- 点开“图灵 1936 年论文”节点，继续问“它和停机问题、冯诺依曼架构、ENIAC 之间是什么关系”

## 5. MVP 的最小闭环

### 5.1 输入

第一版先支持下面两类输入：

- `Markdown` 文本
- 用户手工补充的结构化备注

可选扩展输入，但不作为第一版必做：

- 网页文章
- PDF

### 5.2 输出

第一版必须输出：

- 一条可交互的时间线
- 每个节点的证据面板
- 一个围绕当前节点的对话入口

### 5.3 闭环定义

满足下面条件，就算 MVP 跑通：

1. 用户导入一篇文章。
2. 系统能抽出一批可用事件。
3. 这些事件能形成一条可读主线。
4. 次要信息能以折叠方式挂载。
5. 用户能手工修正错误。
6. 用户能围绕节点继续探索。

---

## 6. 成功标准

MVP 成功不看“模型有多聪明”，而看它是否真的改善研究过程。

### 6.1 定性标准

- 用户能比纯手工整理更快搭出第一版时间线
- 用户能更容易识别主线与支线
- 用户能通过节点追问，补出原文中未显式写出的关键背景
- 用户不会因为 AI 输出缺乏出处而失去信任

### 6.2 定量标准

第一阶段可用下面的轻量指标判断：

- 从导入文章到生成首版时间线，时间控制在 `5 分钟内`
- 从文章中抽出的事件里，用户认为“值得保留”的比例达到 `60%+`
- 节点展开后，`100%` 能看到证据来源
- 用户对时间线做的人工修改集中在“微调”，而不是“推倒重来”

## 7. 核心功能范围

### 7.1 必做

#### A. 导入与切分

- 导入 Markdown 文章
- 按段落或语义块切分内容
- 保留原文位置，便于后续证据回溯

#### B. 结构化抽取

- 抽取时间表达
- 抽取事件
- 抽取人物/机构/概念
- 识别事件与实体的关联

#### C. 时间线生成

- 自动按时间排序
- 区分主线事件和补充信息
- 支持折叠/展开支线
- 支持单点事件和时间区间

#### D. 节点详情与证据

- 查看事件摘要
- 查看关联人物/概念
- 查看证据原文
- 查看来源位置

#### E. 人工修正

- 修改事件标题
- 修改时间
- 修改重要度
- 修改挂载关系
- 合并重复事件

#### F. 节点追问

- 围绕当前事件节点发起 AI 对话
- 对话上下文默认包含该节点及相关证据
- 可把追问结果沉淀回节点备注

### 7.2 可选但非首发

- 关系图视图
- 导图视图
- 自动聚类主题
- 跨文档合并图谱
- 导出公开展示时间线

### 7.3 明确不做

- 多人实时协作
- 评论与权限系统
- 完整项目管理能力
- 通用聊天产品
- 自动网页抓取与大规模知识库同步

## 8. 第一版信息模型

第一版不需要完整 ontology，但至少需要下面几个核心对象。

### 8.1 Source

表示来源材料。

最少字段：

- `id`
- `title`
- `type`：markdown / note / pdf
- `content`
- `created_at`

### 8.2 Evidence

表示一个事件或结论对应的原文证据。

最少字段：

- `id`
- `source_id`
- `quote`
- `start_offset`
- `end_offset`
- `confidence`

### 8.3 Entity

表示人物、机构、概念、论文、系统等对象。

最少字段：

- `id`
- `name`
- `type`
- `aliases`

### 8.4 Event

这是第一版最重要的对象。

最少字段：

- `id`
- `title`
- `summary`
- `time_start`
- `time_end`
- `time_precision`：year / month / day / fuzzy
- `importance`
- `parent_event_id`
- `status`：draft / confirmed / revised

### 8.5 Relation

表示实体和事件之间的关系，或事件之间的关系。

最少字段：

- `id`
- `from_id`
- `to_id`
- `type`
- `confidence`

## 9. 关键产品原则

### 9.1 事件优先

第一版不是“概念卡片优先”，而是“事件优先”。  
时间线是主界面，事件是最核心节点。

### 9.2 证据先于解释

AI 的解释可以错，但证据必须能回溯。  
任何重要节点都应该能看到它从哪一段原文来。

### 9.3 AI 是研究助手，不是裁判

系统要负责：

- 减少机械整理
- 提供候选结构
- 提供追问能力

系统不负责：

- 宣布最终历史结论
- 隐藏不确定性

### 9.4 先收敛到一条主线

第一版的价值不是“信息量最大”，而是“脉络最清楚”。

---

## 10. 建议的交互形态

### 10.1 主界面

第一版主界面建议分成三栏：

1. 左侧：资料结构或来源列表
2. 中间：主时间线
3. 右侧：节点详情 / 证据 / AI 对话

### 10.2 时间线节点

每个节点至少显示：

- 时间
- 标题
- 重要度
- 是否有支线挂载

### 10.3 支线折叠

建议采用“主线展开、支线默认收起”的策略：

- 主线节点直接可见
- 背景说明挂在主线下方
- 点开后再显示补充说明、争议、旁支人物

## 11. 技术边界与实现原则

### 11.1 第一版仍然控制产品范围，但允许底层更超前

MVP 阶段仍然要控制产品范围，但不再要求底层实现也必须保守。

优先原则：

- 先支持 Markdown
- 先生成时间线
- 先保留证据和可编辑性
- 底层对象从第一天就按图谱模型组织

### 11.2 建议的第一版技术方向

结合当前 [技术调研](./01-investigate.md) 和新的架构决策，第一版按下面方向收敛：

- 输入：Markdown
- 抽取：LLM + 规则
- 时间归一化：规则优先，LLM 辅助
- 存储：图谱化信息模型 + 图数据库优先
- 时间线 UI：优先选现成时间轴组件
- 节点追问：围绕当前节点做图邻域上下文注入

换句话说，第一版仍然验证产品闭环，但闭环建立在 `graph-first` 的底层之上。

## 12. 第一阶段验收任务

用当前 [article.md](../../article.md) 作为验收样本。

### 验收要求

1. 从文章中抽取 `20-50` 个事件候选。
2. 能把这些事件按时间排序。
3. 能区分核心事件与补充信息。
4. 每个保留节点都能回看证据原文。
5. 用户可以手工调整至少以下内容：
   - 时间
   - 标题
   - 重要度
   - 父子挂载关系
6. 用户可以点开任一节点继续追问。

### 验收通过标准

如果用户看完这条时间线后，能比直接阅读原文更快回答下面问题，MVP 就有价值：

- AI 发展的主线是怎么展开的？
- 哪些事件是关键拐点？
- 哪些内容只是背景，不应占据主线？
- 某个关键节点和前后节点之间是什么关系？
